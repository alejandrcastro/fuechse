Club Manager

Management system for Füchse Berlin Reinickendorf e.V. · TVBB Berlin

A two-app web system for managing teams, players, matches, availability and lineups. Runs entirely in the browser — no installation, no build step — backed by Supabase.

🔧 Admin Panel

Import PDFs, configure teams, assign players, generate lineups and enforce TVBB blocking rules.

📱 Player App

Mobile-optimized view for players to confirm availability and download calendar events.

📄 PDF Import

Auto-imports teams, players and matches from official TVBB Berlin PDFs — no manual entry.

⚖️ TVBB Rules

Stammkader and Stammmannschaft enforced automatically with real-time blocking alerts.

Login & Access

Two-level access control

Admin Panel — Supabase Auth

The admin panel uses Supabase Authentication. Login credentials are managed in the Supabase dashboard under Authentication → Users. The session persists across page reloads using Supabase's built-in token storage.

✅ Security

Write access to the database is protected by both Supabase Auth and Row Level Security (RLS). Even if someone obtains the publishable API key, they cannot write to the database without a valid authenticated session.

To add or remove admin users, use the Supabase dashboard. No code changes required.

Player App — Shared password

The player app (index.html) uses a simple shared password stored in sessionStorage. This keeps casual visitors out while remaining easy for all players to use. The password is set directly in the HTML file.

The session lasts while the browser tab is open. Closing the tab clears the session.

Row Level Security

With RLS enabled on Supabase, the following policies apply:

OperationWho can perform it
SELECT (read)Anyone (anon + authenticated)
INSERT / UPDATE / DELETEAuthenticated users only
⚠️ Note

Player availability (INSERT to the availability table) is a write operation. If you enable strict RLS, players will not be able to submit their availability from index.html without being authenticated. Consider allowing anonymous inserts on the availability table only, or keeping that table without RLS.

PDF Import

Import data from official TVBB Berlin PDFs

Three PDF types are supported, each with drag & drop or click-to-upload. Always import in this order: 1. ResultReport → 2. Namentliche Meldung → 3. Begegnungen.

1. ResultReport → Teams

Auto-detects all adult teams and their leagues from the Ergebnistabellen PDF. Juvenile categories (U-series, Midcourt) are automatically skipped.

GenderInferred from category (Herren=M, Damen=F)
SeasonAlways "Sommer" — set Winter manually if needed
💡 Safe to re-import

Existing teams are never overwritten. Only new teams are inserted.

2. Namentliche Mannschaftsmeldung → Players

Imports the full player roster with LK, birth year, TVBB ID number, and team assignments. Each player is automatically assigned to their Stammkader team and all higher teams they are eligible for within the same category.

💡 Multiple PDFs

Upload multiple PDFs at once (Ctrl+Click) to merge Herren, Damen and Senioren lists in one step.

ℹ️ Existing players

If a player already exists (matched by IDnr), only their LK is updated if it has changed. Team assignments are added, never removed.

3. Begegnungen → Matches

Imports the full match schedule from the Begegnungen PDF. Juvenile matches (U-series, Midcourt) are filtered out automatically.

⚠️ Teams must exist first

The Begegnungen parser matches each match to an existing team in the database. Import teams from the ResultReport before importing matches.

ℹ️ Team number matters

The team number determines blocking rule calculations. Herren = #1, Herren II = #2, etc. Numbers ≥ 10 are treated as misimported age-group numbers and ignored.

Players

Roster management and team assignments

Player list

Players are sorted by LK and filterable by team, gender, and blocking status. Each player card shows their team chips, LK, birth year, and any active blocking badges (⛔ gesperrt, ⚠ warned).

Team assignment

Each player is automatically assigned to all teams they are eligible for based on their Stammkader position. The assignment modal allows manual overrides and shows current match counts per team.

Match count adjustment

The match count (used for Stammmannschaft calculations) can be incremented or decremented manually in the assignment modal. This is useful for correcting counts after retroactively entering results.

Automatic team expansion

On every load, the system verifies that each player is assigned to all teams they should be eligible for — from their Stammkader team upward (lower-numbered teams). Missing assignments are added automatically.

Matches

Schedule management

Match list

All matches are shown sorted by date with a progress bar showing availability (Ja responses / total squad). Each card shows home/away status, venue, and lineup status.

Quick links

Each match card has a Player View button that opens the availability input overlay directly for that match — useful for capturing availability by phone during a conversation.

Adding matches manually

Use the form at the top to add individual matches. For full schedule imports use the Begegnungen PDF import.

Lineup

Player assignment and validation

Category filter

Filter the match selector by category to quickly find the right match.

Auto-generate

The ⚡ button generates a lineup automatically from players who said Ja, are not blocked, sorted by LK. Singles slots are filled first in LK order, then doubles pairs are formed from the same players.

Player dropdowns

Each slot has a dropdown showing all squad members. Players are grouped by availability status (same order as the availability table) so Ja players appear at the top. A separator line labels each group.

Validation

The lineup is validated in real time:

  • Rank order — singles must be ordered by rank (position 1 must have a lower or equal rank than position 2, etc.). This generally corresponds to LK order but rank takes precedence — a player can be placed above a better-LK player if their rank in the category warrants it. Violations are shown in red.
  • Blocking — assigning a blocked player triggers a red warning banner.
  • Valid — a green banner confirms the lineup is rule-compliant.
ℹ️ Doubles

Doubles pairs are assigned automatically from the singles lineup. The sum of slot numbers in each double must not exceed the sum of the following pair (TVBB rule).

Player App

Mobile-optimized availability confirmation

Flow

One shared password for all players — set in the HTML file by the administrator.

Find yourself

Type your name in the search field. Autocomplete shows matching players from the current roster.

See your matches

All matches across all teams you are eligible for (Spielberechtigt) are shown, sorted by date.

Team filter

Each team chip at the top of the match list acts as a toggle. Tap one or more chips to show only matches for those teams. Tap again to deactivate. The filter resets when you select a different player.

💡 Multi-select

Multiple chips can be active at the same time — useful for players who want to see Herren and Herren 30 together but not Herren II.

Calendar export (.ics)

Each match card has a 📅 .ics button that downloads the match as a calendar event, compatible with Google Calendar, Apple Calendar and Outlook. The event title reflects home/away order: Herren vs TC Blau-Weiss for home matches, TC Blau-Weiss vs Herren for away matches.

Key relationships

player_teams.rank is the player's position within the category's Namentliche Mannschaftsmeldung list. This is independent per category — a player can have rank 3 in Herren and rank 7 in Herren 30.

⚠️ team_number vs age group

The team_number column stores the team sequence number (1, 2, 3…) — not the age group number. A team named "Herren 30 II" has team_number = 2, not 30. Values ≥ 10 indicate a bad import and are ignored by the client.

TVBB Rules

§10 WSpO — exact implementation (April 2024)

§10 Abs. 1 — Stammkader

Definition: Players in each category are ordered by their position in the Namentliche Mannschaftsmeldung. The first N form the Stammkader of team 1, the next N of team 2, and so on. A Stammspieler has no Spielberechtigung for lower-numbered (weaker) teams.

  • N = 6 for Sommer 6er-teams
  • N = 4 for Winter or 4er-teams (Damen 55+, Herren 65+, etc.)
  • Ranks are per category — Herren and Herren 30 have independent rank lists
  • Players can always play in higher-numbered (weaker) teams

Example (Sommer, Herren): Rank 1–6 → Stammkader Herren (blocked in II, III). Rank 7–12 → Stammkader Herren II (blocked in III only). Rank 13–18 → Stammkader Herren III (no restriction).

§10 Abs. 2 — Stammmannschaft

Definition: A player from a lower team may be used as Ersatzspieler in a higher team only once. The rule is evaluated cumulatively: for a given team X, the sum of all matches played in teams with a lower number than X determines eligibility.

  • Sum = 0 → no restriction
  • Sum = 1 → Verwarnung ⚠ (next match would block)
  • Sum ≥ 2 → gesperrt in team X
  • Applies independently per category
  • Does not affect playing in better (lower-numbered) teams

Example: Player plays 1× in Herren and 1× in Herren II. For Herren III: sum of superior matches = 2 → gesperrt. For Herren II: sum = 1 (only Herren) → ⚠ warning.

ℹ️ Both rules are independent

A player can be blocked by Stammkader only, Stammmannschaft only, or both simultaneously. The blocking view shows the source of each restriction.

getPlayerBlockStatus() — logic summary

// For each team X in category:

// Rule 1 — Stammkader
rankInCat = player_teams.rank for any team in this category
skTeam    = walk catTeams subtracting groupSize until rank fits
skBlocked = catTeams where teamNumber > skTeam.number

// Rule 2 — Stammmannschaft
for each team X:
  superiorMatches = sum of matchCounts for all teams with number < X
  if superiorMatches >= 2 → smBlocked.push(X)
  if superiorMatches == 1 → smWarned.push(X)

// A team is blocked if EITHER rule blocks it

PDF Parsers

Technical documentation

PDF.js text items

PDF.js returns document text as a flat list of items. Each item has item.str (text), item.transform[4] (X position), and item.transform[5] (Y from bottom of page).

const x = Math.round(item.transform[4]);
const y = Math.round(viewport.height - item.transform[5]);
// Group rows: key = Math.round(y / 4) * 4  (4px tolerance)
🚫 Critical: cells are whole strings

PDF.js returns each table cell as a single complete string. The string "Füchse Berlin Reinickendorf e.V. III" is one item — not individual words. All parsers use item.str.includes(), not word matching.

ResultReport parser

Four fixed columns at X ≈ 28, 227, 427, 626 px. Header rows contain the full category+league+group string in one item per column.

// Header detection: row contains "Gruppe"
// Adult header regex:
/^(Herren(?:\s+\d+)?|Damen(?:\s+\d+)?)\s+(.+?)\s+Gruppe\s+[A-Z]$/

// Data row: item.str.includes("Reinickendorf")
// Team number from: item.str.match(/e\.V\.\s+(VIII|VII|VI|IV|V|III|II)\s*$/)

// Team header pattern:
/(Herren\s*(?:\d+\s*)?(?:II|III|IV...)?|Damen\s*...)\s*(?:\(4er\))?\s*\|\s*([^\n]+)/gi

// Rank → team assignment:
let remaining = rank;

  if remaining <= teamSize: this is the Stammkader team; break
  remaining -= teamSize
